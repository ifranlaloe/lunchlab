Choosing the Right API Style

A practical system design session using real-world platforms, C4, and explicit trade-off reasoning.
C4 framing REST GraphQL gRPC Webhooks WebSocket SOAP
Why this session

Not all API concepts are equally clear yet

What we want

  • Understand when a certain API style fits.
  • Use real-world examples people recognize instantly.
  • Practice explaining design decisions, not just naming technologies.

How we keep discussions useful

  • Use C4 to define the discussion scope.
  • Stay mostly at Level 1 and Level 2.
  • Only go deeper when it truly helps the design conversation.
C4 cheat sheet

C4 on the left, API styles on the right

The 4 levels of C4

1

System Context

Main question: What is our system, who uses it, and which external systems interact with it?
2

Container

Main question: What are the main applications, services, and data stores, and how do they communicate?
3

Component

Main question: How is one container split into internal building blocks and responsibilities?
4

Code

Main question: How is the solution implemented in code?

API styles we will focus on

C4 focus

We will mainly use Level 1 and Level 2

Level 1 — System Context

  • Shows users and external systems.
  • Answers: what is our system and who interacts with it?
  • Great for identifying external APIs and integrations like payment providers or legacy SOAP systems.
VS

Level 2 — Container

  • Shows the system's main apps, services, and data stores.
  • Answers: how is our system structured internally?
  • Great for discussing REST, GraphQL, WebSocket, and gRPC between containers.
Guardrail

How to keep the discussion in scope

Ask first

  • Which C4 level are we discussing right now?
  • Are we outside the system or inside it?

Stay useful

  • Talk about interactions between boxes.
  • Avoid deep implementation details.

Cheat phrase

“We only talk about communication between boxes, not inside them.”
Case

Real-world example: a food delivery platform

Think of a platform in the space of Uber Eats or Deliveroo. Familiar enough to imagine, rich enough to contain several API styles.
Customer App→ Delivery Platform← Restaurant Portal← Driver App
Requirements

Functional and non-functional requirements

Functional

  • Browse restaurants and menus.
  • Place an order and pay.
  • Restaurant receives and accepts orders.
  • Customer tracks delivery in near real time.
  • Customer receives order status updates.

Non-functional

  • Responsive experience for mobile users.
  • Reliable payment confirmation.
  • Scales during lunch and dinner peaks.
  • Low latency for driver tracking updates.
  • Works with external partners and legacy systems.
Workshop flow

Then we go to the drawing board

1

Identify actors and external systems

Start with Level 1 and draw the system boundary.
2

Identify the main containers

Move to Level 2: apps, services, and important data stores.
3

Draw interactions

Who talks to whom? Which interactions are synchronous, asynchronous, or real-time?
4

Label the API style per interaction

Choose the style that fits the interaction, then explain the trade-off.
Level 1

Start with System Context

Identify

  • Customers
  • Restaurants
  • Drivers
  • Payment provider
  • Notification provider
  • Possibly a legacy partner system

Prompt for the room

  • Which of these are part of our system?
  • Which are outside the boundary?
  • Where do we already expect external API styles like webhooks or SOAP?
Level 2

Then move to Containers

Mobile App → Backend API → Order Service → Payment Service → Tracking Service

Good discussion questions

  • Which containers do we need first?
  • Which container owns which responsibility?
  • Which interactions matter for the user experience?

Important framing

  • gRPC is valid here because it is a choice between containers.
  • We do not need to open up the inside of a service to discuss it.
API mapping

Example mapping of API styles

Client-facing

  • REST for straightforward request-response flows.
  • GraphQL when frontend data needs are variable.
  • WebSocket for live tracking or status streams.

Internal

  • gRPC for fast, strongly typed service-to-service communication.
  • Still a Level 2 container discussion.

External

  • Webhook for payment confirmations or event callbacks.
  • SOAP when integrating with legacy enterprise systems.
Facilitation

Questions to keep asking during the session

About scope

  • Which C4 level is this decision on?
  • Is this inside the system or outside it?
  • Are we still talking between boxes?

About API choice

  • Why does this interaction need this style?
  • What trade-off are we making?
  • What would break if we used a different style?
Takeaway

API styles are not a preference contest

Choose the style that fits the interaction, and explain it within the right C4 scope.
