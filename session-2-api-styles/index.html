LunchLab #2

Choosing the Right API Style

Use C4 as infrastructure cartography to choose API styles at the right architectural zoom level.
Level 1: Regional map Level 2: City map REST GraphQL WebSocket Webhook gRPC SOAP
C4 framing

C4 as infrastructure cartography

Core idea

  • Architecture diagrams are maps.
  • Good maps are designed for a purpose.
  • Different infrastructure questions need different maps.
  • A useful map is not complete; it is purposeful.

Session stance

  • First choose the map scale, then discuss design decisions.
  • C4 gives us shared zoom levels for architecture conversations.
  • Today we focus mostly on Level 1 and Level 2.
C4 levels

C4 levels mapped to infrastructure maps

Level 1 — System Context

Regional infrastructure map

  • Main question: What surrounds and connects to our system?
  • Shows: users, external/partner/SaaS systems, identity and payment providers, system boundary.
  • Omits: internal services, databases, components, code.

Level 2 — Container

City infrastructure map

  • Main question: What are the main places and routes inside our system?
  • Shows: apps, services, databases, message brokers, caches, communication paths.
  • Omits: component internals, classes, functions, implementation detail.

Level 3 — Component

Building systems map

  • Main question: How is one building organized internally?
  • Shows: components/modules, responsibilities, dependencies in one service.
  • Omits: full platform landscape and code-level detail.

Level 4 — Code

Engineering blueprint

  • Main question: How is it constructed in detail?
  • Shows: classes, functions, interfaces, methods, packages, queries, implementation choices.
  • Omits: high-level context and broad relationships.
Level 1

Level 1: Regional infrastructure map

At Level 1, we are not mapping every street. We show the region: who comes into our city, which external places we connect to, and where our border is.

Include on the map

  • Users
  • External systems
  • Partner systems
  • Payment providers
  • Identity providers
  • System boundary

Facilitation + API tie-in

  • Facilitator question: “Is this inside our system boundary, or outside it?”
  • Webhook from a payment provider = external delivery route entering our region.
  • SOAP to a legacy partner = old regulated cross-border railway line.
Level 2

Level 2: City infrastructure map

At Level 2, we are city planners. We identify important buildings and infrastructure inside our city, then draw the roads, bridges, tunnels, and lanes between them.

Include on the map

  • Frontend applications
  • Backend APIs
  • Services
  • Databases
  • Message brokers
  • Caches
  • Communication paths

Facilitation + API tie-in

  • Facilitator question: “Which places need to communicate, and what kind of traffic moves between them?”
  • API styles are the infrastructure choices for moving traffic between places.
API mapping

API styles as infrastructure choices

REST = Normal road

Simple request-response traffic.

GraphQL = Route planner / smart junction

Client asks for exactly the data it needs.

WebSocket = Dedicated open lane

Continuous two-way real-time traffic.

Webhook = Delivery drop-off route

One system delivers an event when something happens.

gRPC = High-speed service tunnel

Fast internal service-to-service traffic.

SOAP = Regulated legacy rail line

Strict, formal, often older enterprise integration.

Facilitation

Keeping the discussion at the right map scale

  • “Before choosing the road type, first agree on the map scale.”
  • “That sounds like a building-systems question. Useful, but for now we are still designing the city infrastructure.”
  • “That belongs on the engineering blueprint. Let’s first agree on the infrastructure map.”
  • “If we are discussing bridge materials, we probably zoomed in too far.”
Scope control

Why this LunchLab stays mostly at Level 1 and Level 2

Main focus now

  • Regional map: boundaries and external connections.
  • City map: internal containers and communication routes.
  • Choose API styles where traffic moves between places.

Useful later

  • Level 3 (Component) helps internal service design.
  • Level 4 (Code) helps implementation and refactoring.
  • Important, but not the main focus of today’s session.
Exercise

Room exercise: map first, API style second

1

Pick the map level

Is this a regional or city infrastructure question?
2

3

4

Takeaway

Cartography first, implementation later

Today we mostly use the regional map and city map. API styles are the road types between places.
Choose map scale first Level 1 + Level 2 now APIs = infrastructure choices Avoid zooming too far too early
