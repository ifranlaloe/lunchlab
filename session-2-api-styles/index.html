Choosing the Right API Style

There is no single best API style. The useful question is: what is the interaction, and which style fits it best?

Why this session

  • Choose API styles with clearer architectural scope.
  • Practice explaining trade-offs, not preferences.
  • Use shared C4 language before solution debates.

What you will get

  • A practical Level 1 + Level 2 decision workflow.
  • Interaction archetypes mapped to API styles.
  • Facilitation prompts to avoid over-zooming.

C4 framing

C4 as infrastructure cartography

Core idea

  • Architecture diagrams are maps.
  • Good maps are designed for a purpose.
  • Different infrastructure questions need different maps.
  • A useful map is not complete; it is purposeful.

Facilitation intent

  • We align C4 language first, before any app example.
  • First choose the map scale, then discuss decisions.
  • Today we focus mostly on Level 1 and Level 2.
C4 naming

Why it is called C4

Context

System boundary and external actors.

Containers

Main runtime building blocks and routes.

Components

Internal modules inside one container.

Code

Implementation-level structures and details.

C4 levels

C4 levels mapped to infrastructure maps

Level 1 — System Context

Regional infrastructure map

  • Main question: What surrounds and connects to our system?
  • Shows: users, external/partner/SaaS systems, identity and payment providers, system boundary.
  • Omits: internal services, databases, components, code.

Level 2 — Container

City infrastructure map

  • Main question: What are the main places and routes inside our system?
  • Shows: apps, services, databases, message brokers, caches, communication paths.
  • Omits: component internals, classes, functions, implementation detail.

Level 3 — Component

Building systems map

  • Main question: How is one building organized internally?
  • Shows: components/modules, responsibilities, dependencies in one service.
  • Omits: full platform landscape and code-level detail.

Level 4 — Code

Engineering blueprint

  • Main question: How is it constructed in detail?
  • Shows: classes, functions, interfaces, methods, packages, queries, implementation choices.
  • Omits: high-level context and broad relationships.
Level 1

Level 1 route modalities and trade-offs

Modalities at the boundary

  • Sea route: high capacity, slower cadence.
  • River route: constrained channels, policy sensitivity.
  • Air route: fast, expensive, strict controls.
  • Truck route: flexible, variable reliability.
  • Rail route: scheduled, formal, governance-heavy.

Decision dimensions

  • Latency expectations
  • Governance and trust controls
  • Reliability and resilience behavior
  • Coupling and contract rigidity
Archetypes

Start with interaction archetypes, then style

Archetype

  • Request/response
  • Streaming real-time
  • Event callback
  • Internal RPC
  • Legacy boundary integration

Likely style candidates

  • REST, GraphQL
  • WebSocket
  • Webhook
  • gRPC
  • SOAP
API mapping

API styles as infrastructure choices

REST = Normal road

Simple request-response traffic.

GraphQL = Route planner / smart junction

Client asks for exactly the data it needs.

WebSocket = Dedicated open lane

Continuous two-way real-time traffic.

Webhook = Delivery drop-off route

One system delivers an event when something happens.

gRPC = High-speed service tunnel

Fast internal service-to-service traffic.

SOAP = Regulated legacy rail line

Strict, formal, often older enterprise integration.

Facilitation

Keeping the discussion at the right map scale

  • “Before choosing the road type, first agree on the map scale.”
  • “That sounds like a building-systems question. Useful, but for now we are still designing the city infrastructure.”
  • “That belongs on the engineering blueprint. Let’s first agree on the infrastructure map.”
  • “If we are discussing bridge materials, we probably zoomed in too far.”
  • “Are we still talking between boxes?”
Exercise

Guided C4 calibration before domain case

Sample decisions to classify

  • “Partner sends status updates to us.”
  • “Service A needs low-latency calls to Service B.”
  • “Retry policy should be exponential backoff.”
  • “Need field-level query flexibility for mobile.”

Prompt bank

  • Which C4 level is this decision on?
  • Are we still talking between boxes?
  • What information is missing at this zoom level?
  • Should we decide this now or park it for Level 3/4?
Takeaway

Cartography first, implementation later

No universal best API style. Fit depends on interaction, boundary, and route constraints.
Choose map scale firstLevel 1 + Level 2 nowAPIs = infrastructure choicesAvoid zooming too far too early
Cheat sheet

API style selection cheat sheet (final reference)

Client query flexibility

Start with REST; use GraphQL when client-specific data shaping is central.

Real-time bidirectional

Use WebSocket when both sides need continuous low-latency updates.

Asynchronous notification

Use Webhook when producer pushes events to consumer endpoints.

Internal low-latency RPC

Use gRPC for typed, high-throughput service-to-service communication.

Legacy regulated integration

Use SOAP when enterprise governance/contracts require it.

Facilitator rule

Decide boundary and container interactions first; park Level 3/4 implementation details.
