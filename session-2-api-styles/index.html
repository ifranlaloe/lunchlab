Case

Case flow

What has to happen in this platform

Main user flow

  • Customers browse restaurants and menus.
  • Customers place and pay for orders.
  • Restaurants receive and accept orders.
  • Drivers pick up and deliver orders.
  • Customers track delivery progress.

Constraints that matter

  • Mobile experience needs to feel responsive.
  • Payment confirmation needs to be reliable.
  • Driver tracking needs low latency.
  • The platform must scale during lunch and dinner peaks.
  • External and legacy integrations are realistic.
API styles

Use the case to explain different interaction types

Client-facing

  • REST for straightforward request-response flows.
  • GraphQL when frontend data needs are variable.
  • WebSocket for live tracking or status streams.

Inside and outside the platform

  • gRPC for fast, typed communication between services.
  • Webhook for payment confirmations and event callbacks.
  • SOAP for external legacy or enterprise integrations.
API mapping

Example mapping in the food delivery case

Customer side

  • Customer app to backend: REST or GraphQL.
  • Tracking updates to the app: WebSocket.

Platform side

  • Order service to dispatch service: gRPC.
  • Fast and strongly typed, still a container-level choice.

External side

  • Payment provider callback: Webhook.
  • Legacy partner integration: SOAP.
C4

Why it is called C4

The name

  • Context
  • Containers
  • Components
  • Code

Why this helps

  • The name already tells you the four abstraction levels.
  • It gives the room a shared language for architectural scope.
  • For this session we only need the first two levels.
C4 zooming

Think of C4 as zooming into WhatsApp

Same system, different levels of detail. Each step zooms into the previous box.
C4 focus

We stay at Context and Containers

Level 1 — System Context

  • What is our system?
  • Who uses it?
  • Which external systems interact with it?
VS

Level 2 — Container

  • Which apps, services, and data stores exist?
  • How do those containers communicate?
  • Where do API styles fit between those boxes?
C4 guardrails

How to keep the discussion in scope

Ask first

  • Which C4 level are we on?
  • Are we outside the system or inside it?

Keep it clean

  • Talk about communication between boxes.
  • Avoid implementation details inside a service.

Useful examples

  • SOAP fits naturally at Level 1.
  • gRPC still fits cleanly at Level 2.
Facilitation

The room rule for this session

We only talk about communication between boxes, not inside them.

Why this matters

  • It prevents the discussion from dropping into implementation too early.
  • It keeps API choices connected to architecture and interactions.

What it clarifies

  • SOAP can be discussed as an external system integration.
  • gRPC can be discussed as a container-to-container communication choice.

Prompts

Questions to keep asking during the room exercise

About scope

  • Which C4 level is this decision on?
  • Are we still talking between boxes?
  • Is this inside the system or outside it?

About trade-offs

  • Why does this interaction need this style?
  • What trade-off are we making?
  • What changes if traffic or latency changes?
Twist cards

Add a constraint and revisit the API choice

Real-time update

  • The customer now expects live status updates.
  • Does the API choice still fit?

Latency target

  • A service interaction must stay below 100 ms.
  • What changes at container level?

Legacy partner

  • A partner system only exposes SOAP.
  • Where does that sit in the model?
Takeaway

Cheat sheet

Quick reference for the session

API styles

  • REST: straightforward request-response
  • GraphQL: flexible frontend data shaping
  • WebSocket: real-time updates
  • Webhook: async callback from another system
  • gRPC: container-to-container communication
  • SOAP: legacy or enterprise integration

C4 framing

  • Context: users and external systems
  • Containers: apps and services inside the system
  • Components: internal building blocks inside one container
  • Code: implementation details in code
  • Stay at Level 1 and Level 2 for this session
  • Talk about communication between boxes
  • Explain the trade-off behind each choice
